## Action plan — concrete next steps

### Step 2 (off-chip): empty-trace forensics

Investigate why `grade-school.t32` and `ini-parser.t32` cells from c1 produced 0-iteration runs. Distinguishing question: is `join_status: orphan_transcript_record` + `total iterations: 0` a harness/registry-join symptom, or a genuine "model emits no tool call on turn 1" terminal error?

Concrete things to check:
- Compare against `alphametics.t32` (c1) which had 1 iter — same `orphan_transcript_record` status but with content. Is the iter count the differentiator, or are they the same shape?
- Check `host/test/lib/run_row.js` for how `terminal_status='error'` is assigned when iteration count is 0.
- Check `host/test/lib/registry.js` and `host/test/lib/claw.js` join logic for any path that emits a row with empty iterations on a transcript-fetch failure.
- Look at the broader sweep log [`logs/OVERNIGHT-SCREEN-explore-c1-20260502-1735.md`](../base/...) (or wherever the cycle-1 log lives — committed at `2ecb3c0` in `host/test/logs/`) for the rep where these tests ran; look for stderr/exit-code anomalies.

Output target: a 1-paragraph diagnosis appended to this file under §"Step 2 findings", plus one of:
- "Re-pilot is the right move (c3 t32 N=3 on these two cells)" — proceed to step 3.
- "Harness fix X first" — defer c3 until X lands.

### Step 3 (chip): cycle-3 re-pilot

Assuming step 2 says re-pilot:

```
TIER_EVAL_FILTER="grade-school ini-parser" \
EVAL_REPS=3 \
host/test/scripts/explore-cycle.sh 3 32
```

Outputs land at `host/test/docs/difficulty-pack/explore/c3/summary.md` and snapshots/. Expected runtime ~20-40min serial including plist swap + cold-load (chip-bound).

### Step 4 (off-chip, parallel with step 3): B tweaks

For `two-bucket` and `word-search`:
- Inspect the verifier `node:assert/strict` calls — wrap any opaque assertions with messages that include *what the model returned* and *what was expected*, formatted so the model can read them in `bash node verify.js` output.
- Check the prompt for return-shape ambiguity (e.g., array of pairs vs object of arrays for word-search; pour-action object schema for two-bucket).
- After edits, re-pilot at N=3 t16 only via `TIER_EVAL_FILTER="two-bucket word-search"` — small chip cost.

### Step 5: reconvene

After c3 lands, update [`PLAN.md`](PLAN.md) verification gate count and decide on D-pool dip vs gate-closed.
